Design Patterns In Plain English

A

Layered / Ports-and-Adapters style

Commands and filesystem are outside; the domain logic sits in the center. This reduces accidental coupling.

B

Use-Case / Application Service pattern

Each command calls a specific workflow object or function instead of reaching into random helpers.

C

Value Objects / Normalized records

Represent skills, packages, build-state entries, and dependency results in one canonical shape.

D

Repository-style access

One place to load and persist skill/build/install state instead of open-coded file access everywhere.

E

Orchestration, not god modules

Use cases coordinate smaller modules. No single file should own every rule and every side effect.
