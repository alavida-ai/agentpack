## What you'll learn

You'll walk through the full publish-to-install cycle: publishing a skill package, configuring a consumer repo to pull from the registry, and using agentpack to install, inspect, update, and remove exported skills.

## Prerequisites

- Completed [The Skill Graph](/skill-graph)
- A GitHub Packages-enabled repository
- Authenticated with `agentpack auth login` (see [Authentication](/authentication))

## Publishing a skill package

<Steps>

<Step title="Validate before publishing">

Always validate before you publish. This ensures the package contract and exported skills are structurally sound.

```bash
agentpack skills validate domains/brand/skills/copywriting
```

```
Package: @acme/brand-copywriting
Status: valid
```

Validation checks that exported skills are declared under `package.json.agentpack.skills`, exported paths resolve, compiler-mode skill imports align with package dependencies, and package metadata is complete.

</Step>

<Step title="Publish with npm">

agentpack does not replace `npm publish`. Publishing uses standard npm tooling:

```bash
cd domains/brand/skills/copywriting
npm publish
```

The package is now on GitHub Packages. Exported `SKILL.md` files and package metadata are published; source knowledge files stay in the authoring repo.

<Note>
For CI publishing, use a workflow that writes `.npmrc` from `GITHUB_PACKAGES_TOKEN` before running `npm publish`.
</Note>

</Step>

</Steps>

## Configuring a consumer repo

A consumer repo is any repo that uses published skills but does not author them. The fastest way to configure it:

```bash
agentpack auth login
agentpack auth status --verify
```

This configures the scope, registry, and token in one step. See [Authentication](/authentication) for details, manual `.npmrc` setup, and CI configuration.

Verify the repo-level wiring:

```bash
agentpack skills registry
```

```
Scope: @acme
Configured: true
Registry: https://npm.pkg.github.com
Always Auth: true
Path: .npmrc
Auth: environment variable reference
Auth Key: GITHUB_PACKAGES_TOKEN
```

## Installing skills in a consumer repo

<Steps>

<Step title="Install a skill package">

```bash
agentpack skills install @acme/brand-copywriting
```

agentpack calls npm to resolve the full dependency graph, then materializes every exported skill in the resolved package closure into `.claude/skills/` and `.agents/skills/`.

</Step>

<Step title="Verify the installation">

```bash
agentpack skills env
```

Use this to inspect what package versions are installed and which exported skills were materialized into the runtime directories.

</Step>

<Step title="Check overall status">

```bash
agentpack skills status
```

Use this to check outdated packages, missing dependencies, deprecated packages, and registry configuration in one pass.

</Step>

</Steps>

## Managing installed skills

<Steps>

<Step title="Check for updates">

```bash
agentpack skills outdated
```

</Step>

<Step title="Update a package">

```bash
agentpack skills install @acme/brand-copywriting
```

Reinstall to pick up the latest version. agentpack re-resolves the package graph and re-materializes exported skills.

</Step>

<Step title="Uninstall a package">

```bash
agentpack skills uninstall @acme/brand-copywriting
```

agentpack removes the package from the direct install set, rebuilds runtime state, and cleans up materialized links that are no longer needed.

</Step>

</Steps>

## Summary

| Step | Command | Who |
|---|---|---|
| Validate | `agentpack skills validate <path>` | Skill author |
| Publish | `npm publish` | Skill author |
| Configure registry | Create `.npmrc` | Consumer team (once) |
| Install | `agentpack skills install <package>` | Consumer |
| Check health | `agentpack skills status` | Consumer |
| Check updates | `agentpack skills outdated` | Consumer |
| Update | `agentpack skills install <package>` | Consumer |
| Remove | `agentpack skills uninstall <package>` | Consumer |

agentpack handles the agent-specific parts: skill resolution, materialization, runtime state, and provenance. npm handles versioning, package resolution, and registry distribution.

## Publisher package.json

Your skill package's `package.json` must declare the exported skills and point at GitHub Packages:

```json package.json
{
  "name": "@acme/brand-copywriting",
  "version": "1.2.0",
  "description": "Brand copywriting package.",
  "files": ["skills/"],
  "agentpack": {
    "skills": {
      "brand-copywriting": {
        "path": "skills/brand-copywriting/SKILL.md"
      }
    }
  },
  "repository": {
    "type": "git",
    "url": "git+https://github.com/acme-corp/knowledge-base.git"
  },
  "publishConfig": {
    "registry": "https://npm.pkg.github.com"
  }
}
```

See [package.json Reference](/schema-package-json) for all fields.

## CI publishing

For automated publishing in GitHub Actions:

```yaml .github/workflows/publish.yml
- name: Publish skill package
  env:
    NODE_AUTH_TOKEN: ${{ secrets.GITHUB_PACKAGES_TOKEN }}
  run: |
    npm publish -w domains/brand/skills/copywriting
```

For automated install in CI:

```yaml .github/workflows/install-skills.yml
- name: Install skills
  env:
    NODE_AUTH_TOKEN: ${{ secrets.GITHUB_PACKAGES_TOKEN }}
  run: |
    agentpack skills install @acme/brand-copywriting
```

<Note>
  `NODE_AUTH_TOKEN` is the environment variable npm reads for registry auth. Set the `GITHUB_PACKAGES_TOKEN` secret to a GitHub PAT with `read:packages` and `write:packages` scopes. See [Authentication](/authentication) for details.
</Note>

See [Authentication](/authentication) and [CI Integration](/ci-integration) for complete workflow examples.
