Architecture: Current vs Cleaner Target

A visual model for making agentpack easier to understand and maintain

Current Shape
Commands

skills

Validate, stale, install, env, dependencies, dev, unlink, status.

↓
Mixed Core

src/lib/skills.js

Authoring rules, dependency graph, install state, runtime linking, provenance, stale detection, env reporting.

↔
Parallel Logic

plugin + plugins.js

Reuses some skill logic, but still owns its own traversal and workflow shape.

Result: the domain idea is simple, but the implementation feels blurry and brittle.
Cleaner Target
Core Domain

Packaged Skill Lifecycle

The tool’s main truth: what a packaged skill is, what it depends on, where it came from, and where it is in its lifecycle.

↓
Thin Command Surface

skills

Calls lifecycle services for inspect, validate, stale, install, env.

Composition Layer

plugin

Builds and vendors plugin artifacts using the same skill lifecycle and dependency rules.

↓
Shared Modules

Metadata + Graph + Provenance

Parse skill metadata once, resolve dependencies once, record build-state once.

Separate Effects

Install + Link + Bundle

Filesystem and distribution actions are isolated from pure domain logic.

Result: one obvious center, smaller modules, less accidental duplication.

What “Plugins Reuse The Skill Core” Actually Means

A

Good Reuse

Plugin code asks the skill core for parsed metadata, dependency closure, lifecycle status, and provenance records.

B

Bad Reuse

Plugin code imports a giant utility file and becomes coupled to unrelated install/dev/output logic.

Suggested Module Boundaries

Skill Model

Parse skill/package metadata and normalize paths.

Skill Graph

Resolve requires, reverse dependencies, closure, affected state.

Provenance

Build-state records, source hashes, stale comparison.

Runtime Materialization

Install state, local links, reconciliation into agent-visible directories.

Plugin Packaging

Stage, vendor, emit bundled plugin artifacts using the shared skill services.
