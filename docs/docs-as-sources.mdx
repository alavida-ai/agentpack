---
title: "Docs as Sources"
description: "The core pattern that makes agentpack powerful: compile your team's knowledge docs into tracked, versioned skills."
---

Your team already maintains knowledge in markdown: brand guidelines, research methodology, style guides, onboarding docs. That knowledge is the truth. Skills should be compiled from it, not copy-pasted alongside it.

## The pattern

```
Source docs (truth)  →  SKILL.md (compiled artifact)  →  npm package (distributed)
```

1. **Keep truth in docs.** Your team writes and maintains knowledge in markdown files. These are the source of truth -- the files people actually update.
2. **Bind them in the `agentpack` block.** Each SKILL.md declares which knowledge docs it was built from with explicit `source` bindings.
3. **Package as a skill.** The skill becomes a versioned npm package that any team can install.

When a source doc changes, agentpack knows. It compares the current file hash against the hash recorded at the last validation and marks the skill as **stale**. You always know which skills need updating and why.

## Why this matters

Without source tracking, knowledge drifts silently. Someone updates the brand guidelines, but the copywriting skill still teaches agents the old tone. Nobody notices until a customer-facing draft sounds wrong.

With explicit source bindings, the drift is visible:

```
domains/brand/knowledge/tone-of-voice.md  (changed last Tuesday)
        |
        v
@acme/brand-copywriting                   (stale — source hash mismatch)
        |
        v
@acme/landing-page-writer                 (affected — depends on stale skill)
```

Every downstream skill that depends on stale knowledge gets flagged. You can see the full blast radius before anything goes wrong.

## Walkthrough: brand guidelines to published skill

### Step 1: Write the knowledge doc

Your content team maintains a tone-of-voice guide. This is the source doc -- it lives in your repo and gets updated through normal pull requests.

```markdown
# Tone of Voice

- Warm but direct. No hedging.
- Active voice always. "We built this" not "This was built by us."
- Say "you" not "the user."
- No marketing superlatives.
```

### Step 2: Create the skill

A skill author reads the knowledge doc and writes a SKILL.md that distills it into agent instructions. The `agentpack` block records where the knowledge came from.

````markdown
---
name: brand-copywriting
description: Write brand-consistent copy using company tone and voice guidelines.
---

```agentpack
source tone = "domains/brand/knowledge/tone-of-voice.md"
```

Apply [tone guidance](source:tone){context="primary source of tone constraints for customer-facing copy"}.
````

### Step 3: Validate and publish

```bash
agentpack publish validate domains/brand/skills/copywriting
npm version patch -w domains/brand/skills/copywriting
npm publish -w domains/brand/skills/copywriting
```

Validation records the source hash. From this point forward, agentpack can detect when `tone-of-voice.md` changes.

### Step 4: Source doc changes

Three weeks later, the content team updates the tone-of-voice guide to add a new rule about emoji usage. They merge the PR. The knowledge doc is now ahead of the skill.

### Step 5: Staleness is detected

```bash
agentpack author stale
```

```
Stale Skills: 1

@acme/brand-copywriting
  Path: domains/brand/skills/copywriting
  Changed Sources: 1
    - domains/brand/knowledge/tone-of-voice.md
```

The skill author now knows to update the copywriting skill to reflect the new emoji policy, re-validate, and publish a new version.

## Contrast with the old way

| | Without agentpack | With agentpack |
|---|---|---|
| Knowledge location | Copy-pasted into skill files | Lives in source docs, referenced by path |
| Change detection | None -- silent drift | Automatic hash comparison on every `stale` check |
| Update workflow | "Did someone update that prompt?" | `agentpack author stale` shows exactly what changed |
| Downstream impact | Unknown | Dependency graph shows every affected skill |
| Audit trail | None | `compiled.json` records the semantic graph and bound source hashes for the current build |

## Tips

<Tip>
  Commit `.agentpack/compiled.json` when you want compiled semantic state reviewed or shared across teammates or CI.
</Tip>

<Note>
  `source` binding paths are relative to the repository root, not the skill directory. This means a single knowledge doc can be a source for multiple skills across different directories.
</Note>

<Tip>
  Run `agentpack author stale` in CI on every pull request. If a PR modifies a knowledge doc, CI will flag every skill that depends on it -- before the PR merges.
</Tip>
