## Purpose

Define what needs to be true before `agentpack` should be treated as production-ready for daily use in a real monorepo.

## Current State

The CLI is functionally ready for controlled real use:

- fixture suite is green
- one real-repo live validation path is green
- the current stories are covered at the intended boundary

The main remaining work is operational, not architectural.

## What You Can Use Today

Today you can safely use the CLI for:

- skill inspection
- stale detection
- package validation
- install and uninstall with rematerialization
- environment inspection
- outdated visibility
- missing dependency visibility
- registry inspection
- dependency graph inspection

## Recommended Live Smoke Test

Run this against your real monorepo before relying on the CLI day to day:

```bash
cd /path/to/agentpack
npm run smoke:monorepo -- \
  --repo /path/to/your/monorepo \
  --inspect-target domains/value/workbenches/consumer/website-dev/skills/copywriting \
  --install-target /path/to/your/monorepo/workspace/active/architecture/intent-adoption/spike/packages/value-copywriting \
  --touch-source domains/value/knowledge/value-proposition.md
```

The smoke script will:

1. regenerate `.agentpack/catalog.json` and `.agentpack/build-state.json`
2. run `skills inspect`
3. run `skills validate`
4. run `skills install`
5. run `skills env`
6. run `skills missing`
7. run `skills dependencies`
8. run `skills status`
9. optionally run `skills stale` after touching one real source file
10. run `skills uninstall`
11. clean runtime artifacts

## What Still Needs To Be True For Real Publishing

The CLI is ready to support publishing, but publishing itself is not yet production-wired.

You still need:

- real publishable skill package locations, not just spike packages
- `private: true` removed from publishable skill packages
- real semver dependency ranges instead of local `file:` dependencies
- repo or package `.npmrc` for `@alavida:registry=https://npm.pkg.github.com`
- GitHub Packages auth for local use and CI
- CI workflow that runs `npm publish` for changed skill packages

## Minimum Publish Pipeline

For GitHub Packages, the minimum production path is:

1. validate package with `agentpack skills validate`
2. bump version with `npm version patch|minor|major`
3. publish with `npm publish`
4. let consumer repos update with `agentpack skills install <package>`

## Strong Recommendation

Before calling the system fully production-ready, add:

- a CI publish workflow for changed `@alavida/*` skill packages
- at least one real published private package
- one consumer repo proving install from GitHub Packages rather than local paths

At that point, the system stops being only architecture-proven and becomes distribution-proven.
