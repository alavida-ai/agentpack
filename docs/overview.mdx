## Purpose

Build `agentpack` as the package/runtime layer for agent skills and plugins.

The old CLI is treated as legacy. The replacement is not an incremental extension of that lifecycle.

## Decision

agentpack should implement the skills lifecycle as a thin layer on top of npm and the Agent Skills filesystem model.

Core decisions:

- `skill = package`
- `SKILL.md` is the authored skill artifact
- `package.json` is the package and distribution artifact
- plugins are workbenches that can vendor standalone skill packages into self-contained artifacts
- workbench external dependency roots are inferred from local workbench skills' `requires`
- installs are repo-only
- npm is the resolver of truth
- agentpack owns:
  - source-aware stale detection
  - generated authoring metadata
  - repo-local runtime install state
  - materialization into `.claude/skills/` and `.agents/skills/`
  - plugin bundle graph inspection, validation, build, and dev-watch workflow

## Lifecycle

### Authoring

- domain knowledge is the source of truth
- skills and reports are self-contained compiled artifacts
- `metadata.sources` in `SKILL.md` define provenance and stale relationships
- `requires` in `SKILL.md` define direct skill dependencies
- `agentpack skills dev` links one packaged skill into `.claude/skills/` and `.agents/skills/` for local runtime testing
- source-backed authoring commands resolve `metadata.sources` relative to the current repo root

### Distribution

- each skill is published as its own npm package
- private packages should target GitHub Packages first
- consumer repos should route `@alavida` to `https://npm.pkg.github.com`
- package dependencies are compiled from `requires`
- semver follows normal npm rules

### Runtime

- a workbench plugin provides local workbench skills and runtime shell
- agentpack inspects those local skills
- agentpack unions their direct `requires`
- npm resolves the full transitive package graph
- agentpack materializes the resolved packages into repo-local skill directories

### Bundled Plugins

- local plugin skill `requires` remain the dependency truth
- plugin packages can vendor required standalone skill packages at publish time
- bundled provenance is recorded in `.claude-plugin/bundled-skills.json`
- `agentpack plugin build` writes the self-contained artifact under `.agentpack/dist/plugins/<plugin-name>/`
- `agentpack plugin dev` keeps that artifact rebuilt as the source tree changes

## Docs-As-Sources

The clean pattern is:

1. keep durable truth in domain docs or process docs
2. point `metadata.sources` at those files
3. package the behavior as a standalone skill
4. optionally wrap it with a plugin-local skill that declares `requires`

That keeps docs as the source of truth and keeps the runtime skill as the compiled agent-facing artifact.

## Non-Goals

- agentpack does not rewrite stale skills automatically
- agentpack does not embed agentic reasoning into the CLI
- agentpack does not replace npm as the package-resolution layer
- agentpack does not use plugin boundaries as the dependency model
- agentpack does not centralize all state into one large metadata file

## Current Scope

Implemented in the current repo:

- packaged-skill inspect, validate, dev, unlink, stale, install, env, outdated, dependencies, registry, status, missing, and uninstall
- plugin inspect, validate, build, and dev

Still out of scope:

- publish orchestration
- marketplace orchestration
- dashboards
- user-global installs
