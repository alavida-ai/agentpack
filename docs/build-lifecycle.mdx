## Purpose

Define the unified build and development lifecycle for both skills and plugins in agentpack.

Skills and plugins are different artifact types with a shared foundation:

- a **skill package** is a reusable capability artifact (like a library)
- a **plugin package** is a deployable runtime shell for Claude Code (like an app)

Both need to be authored, tested locally, validated, published, and made discoverable to agents at runtime.

## The Compiler Model

agentpack is a compiler toolchain for agent artifacts.

| Compiler concept | Code (TypeScript) | agentpack |
|---|---|---|
| Source files | `.ts` files | knowledge `.md` files (sources) |
| Compilation | `tsc` transforms source | agent reads sources, writes `SKILL.md` |
| Import resolution | scan `import` statements | scan `requires` / body references |
| Dependency manifest | `package.json` (from imports) | `package.json` (from `requires`) |
| Lint / type-check | `tsc --noEmit`, eslint | `agentpack skills validate` |
| Build metadata | `.tsbuildinfo`, sourcemaps | `build-state.json` (source hashes) |
| Stale detection | "source changed, rebuild" | `agentpack skills stale` |
| Publish artifact | npm package | npm package |
| Install | `npm install` / `bun add` | `agentpack skills install` |
| Module resolution | Node resolves from `node_modules/` | materialization into `.claude/skills/` |

The "compiler" for skills is an agent (or human), not a deterministic transform. But the lifecycle around it — dependency resolution, staleness, validation, packaging, distribution — follows the same pipeline every compiler needs.

## Dependency Sync: The `go mod tidy` Model

`SKILL.md.requires` is the single source of truth for skill dependencies. `package.json.dependencies` is always a compiled mirror — fully derived, never manually maintained.

Every command that touches dependencies performs a total bidirectional sync:

- add a `requires` entry → dependency appears in `package.json`
- remove a `requires` entry → dependency disappears from `package.json`

This follows the Go modules model:

| Go | agentpack |
|---|---|
| source `import` statements | `SKILL.md.requires` |
| `go.mod` (compiled from imports) | `package.json.dependencies` (compiled from requires) |
| `go mod tidy` (adds missing, removes unused) | dep sync inside `dev`, `validate`, `plugin build` |

The sync is not a separate command. It happens automatically inside every command that needs it. The developer never thinks about it.

### How the sync works

```
1. Read SKILL.md.requires          → ["@alavida/methodology-gary-provost"]
2. Read package.json.dependencies  → {"@alavida/methodology-gary-provost": "^1.0.0", "@alavida/old-dep": "^2.0.0"}
3. Add anything in requires but missing from dependencies
4. Remove anything in dependencies that is no longer in requires
5. Write package.json
```

Result: `package.json.dependencies` always exactly mirrors `SKILL.md.requires`. No drift, no stale deps, no manual management.

## Two Artifact Types, Two Build Paths

### Skills: libraries

The source directory IS the publishable artifact. No bundling needed. The mechanical step — syncing `requires` into `package.json.dependencies` — happens automatically inside `dev` and `validate`.

```
SKILL.md (authored)
  ↓  requires: [@alavida/methodology-gary-provost]
package.json (dependencies auto-synced from requires)
  ↓  npm publish
@alavida/value-copywriting on GitHub Packages
  ↓  agentpack skills install (in consumer repo)
.claude/skills/value-copywriting (materialized, discoverable)
```

### Plugins: deployable apps

The source directory is NOT the final artifact. Plugins need standalone skill packages vendored in to produce a self-contained release artifact.

```
plugin source/
├── skills/ (local plugin skills with requires)
├── hooks/, templates/, MCPs
└── package.json
  ↓  agentpack plugin build (syncs deps + resolves + vendors standalone skills)
.agentpack/out/plugins/website-dev/ (built artifact)
  ↓  npm publish
@alavida/plugin-website-dev on GitHub Packages
  ↓  Claude installs plugin
Claude plugin cache (complete, all skills included)
```

## Developer Experience

### Two commands that matter

The developer cares about two things: **test it** and **ship it**.

**`dev`** = I want to test this locally in Claude Code.

**`validate`** = I want to ship this.

Both handle dependency sync automatically. Neither requires the other. Use whichever fits your workflow.

### Skill: test it first, then ship

```bash
# 1. Write SKILL.md with requires

# 2. Test locally
agentpack skills dev domains/value/skills/copywriting
#    → syncs requires into package.json.dependencies (adds AND removes)
#    → links skill into .claude/skills/ so Claude discovers it
#    → Claude sees the skill, you iterate

# 3. Ship it
agentpack skills validate domains/value/skills/copywriting
#    → syncs deps again (in case you changed requires during dev)
#    → validates everything: SKILL.md, sources, package metadata
#    → tells you it's ready
npm publish domains/value/skills/copywriting
```

### Skill: just ship it

```bash
# 1. Write SKILL.md with requires

# 2. Ship it
agentpack skills validate domains/value/skills/copywriting
#    → syncs requires into package.json.dependencies
#    → validates everything
npm publish domains/value/skills/copywriting
```

No `dev` step needed. Validate syncs deps and checks everything in one pass.

### Plugin: build, test, ship

```bash
# 1. Write plugin: local skills, hooks, templates, MCP config

# 2. Build the plugin
agentpack plugin build plugins/website-dev
#    → syncs deps for all local skills
#    → resolves direct and transitive standalone skill closure
#    → vendors standalone skill packages into build output
#    → writes bundled provenance
#    → produces .agentpack/out/plugins/website-dev/

# 3. Test with the built artifact
claude --plugin-dir .agentpack/out/plugins/website-dev
#    → Claude sees the complete plugin exactly as consumers will

# 4. Ship it
agentpack plugin validate plugins/website-dev
npm publish .agentpack/out/plugins/website-dev
```

### Watch mode for fast iteration

```bash
# Skills: link + watch source changes
agentpack skills dev domains/value/skills/copywriting
#    → re-syncs deps when SKILL.md changes

# Plugins: build + watch source changes
agentpack plugin dev plugins/website-dev
#    → rebuilds .agentpack/out/ when source changes
#    → claude --plugin-dir stays pointed at the output
```

## Runtime Discoverability

Both skills and plugins must be discoverable by Claude Code at runtime. The mechanism differs by artifact type.

### Skills at runtime

Skills are discovered from `.claude/skills/` and `.agents/skills/`.

| Context | Command | Result |
|---|---|---|
| Dev (local testing) | `agentpack skills dev <path>` | symlinks source to `.claude/skills/` |
| Production (consumer repo) | `agentpack skills install <package>` | npm install → materialize to `.claude/skills/` |

Both end up in the same place. The agent discovers the skill the same way regardless of how it got there.

### Plugins at runtime

Plugins are discovered from Claude's plugin cache.

| Context | Command | Result |
|---|---|---|
| Dev (local testing) | `claude --plugin-dir .agentpack/out/...` | Claude loads the built artifact |
| Production (consumer) | Claude installs published plugin | plugin arrives in cache, complete |

### Why materialization matters

In Node.js, `require('lodash')` works because Node resolves it from `node_modules/`. The package being in `node_modules/` is what makes it usable.

For agents, a skill in `.claude/skills/` is what makes it discoverable. Without materialization, the skill sits in `node_modules/` where Claude cannot see it. Materialization is the agent equivalent of module resolution.

## What Bundling Actually Is

Plugin bundling is not ordinary npm runtime dependency resolution.

Normal npm: package installs, dependencies install separately in a graph.

Bundled plugin: npm is used as build-time package acquisition. agentpack vendors the required standalone skill artifacts into the plugin package. One released plugin artifact contains everything.

This is the same pattern as:

- standalone skills = libraries
- plugin package = deployable app that bundles its dependencies

Libraries remain authored and versioned independently. The deployable artifact bundles what it needs.

### Skills (no build output — source IS the artifact)

```

```

### Plugins (build produces a new artifact)

```

│   └── bundled-skills.json              ← vendoring provenance

│   ├── value-proof-points/SKILL.md      ← vendored standalone (direct)

├── hooks/
├── templates/
└── package.json
```

## Dev vs Release Testing

The build model eliminates the dev/release gap by ensuring dev always tests the same artifact shape as release.

For skills: the source IS the artifact, so there is no gap.

For plugins: `agentpack plugin build` produces the artifact, `claude --plugin-dir` tests it. Publishing ships the same artifact. Dev and release are identical.

The SDLC must still distinguish:

- **source-mode testing** — rapid iteration during authoring
- **built-artifact testing** — release certification

Built-artifact testing is required before publishing. Source-mode testing is acceptable during active development when fast iteration matters more than artifact fidelity.

## Marketplace References

The marketplace should reference published npm packages, not local build output.

```
.agentpack/out/plugins/website-dev/     ← local build artifact (like dist/)
         ↓ npm publish
@alavida/plugin-website-dev@1.0.0      ← published package (marketplace references this)
         ↓ claude plugin install
~/.claude/plugins/cache/...             ← consumer runtime
```

The `.agentpack/out/` directory is a local `dist/` folder — what you test with and publish from. It is not the distribution endpoint.

## Future Possibilities

### Body-derived dependencies

Currently `requires` is declared in SKILL.md frontmatter. A future enhancement could scan the body for `@scope/package` references and derive requires automatically, similar to how bundlers scan `import` statements:

```markdown
When writing copy, apply the principles from @alavida/methodology-gary-provost
for rhythmic sentence variation.
```

This would eliminate the manual `requires` declaration entirely. The body reference becomes both the usage and the dependency declaration.

### Formal import syntax

If body scanning proves too ambiguous for packages outside the `@alavida` scope, a light syntax marker could make references unambiguous:

```markdown
Apply the principles from `@use @alavida/methodology-gary-provost` for rhythm
and `@use @other-org/tone-framework` for voice.
```

agentpack would scan for `@use` directives — working for any scope, any registry, with zero false positives. This is the skill equivalent of an `import` statement.
