Your team's knowledge is trapped in docs that agents can't reliably use.

Someone writes a great process doc. An engineer copy-pastes it into a SKILL.md. It works — until the doc changes and nobody updates the skill. Another team needs the same skill, so they copy it again. Now there are three versions, none of them current, and no way to tell which one your agent is actually running.

**agentpack fixes this.** It treats agent skills like software packages — authored from source docs, tracked for staleness, versioned with semver, and distributed through npm. When a source doc changes, you know which skills are affected. When you publish an update, every team gets it.

## How it works

agentpack follows a compiler model. Your knowledge docs are source files. Skills are compiled artifacts. Packages are the distributable output.

{/* Screenshot: the skill graph dashboard showing 4-5 nodes with dependency edges, at least one node marked "stale" in orange and one marked "current" in green */}

<Steps>
  <Step title="Knowledge docs are your source of truth">
    Your team maintains process docs, methodology guides, and domain knowledge in markdown. These are the **sources** — the single place where truth lives.
  </Step>
  <Step title="Skills are compiled from sources">
    A SKILL.md file declares which source docs it was built from and which other skills it depends on. It is the agent-facing artifact — the compiled output of your knowledge.
  </Step>
  <Step title="Packages make skills distributable">
    Each skill becomes its own npm package with a version, dependencies, and a registry. Teams install exactly the skills they need, and npm handles the dependency graph.
  </Step>
  <Step title="Agents discover skills at runtime">
    Installed skills are materialized into `.claude/skills/` and `.agents/skills/` where agent runtimes find them. Every agent on your team runs the same versioned playbook.
  </Step>
</Steps>

## The dashboard

When you run `agentpack skills dev`, a visual workbench opens in your browser. It renders the skill graph with D3 — showing dependency edges, source provenance links, and lifecycle status for every node.

<CardGroup cols={2}>
  <Card title="Skill graph" icon="diagram-project">
    See every skill and its dependencies as an interactive DAG. Nodes are colored by status: current, stale, or affected by upstream changes.
  </Card>
  <Card title="Inspector panel" icon="magnifying-glass">
    Click any node to see its sources, requires, package version, and validation state. Run actions like validate, refresh, and check-stale directly from the panel.
  </Card>
</CardGroup>

<Note>
  The dashboard is a visibility tool for humans. Agents use the CLI commands directly — they never interact with the dashboard.
</Note>

## Staleness you can see

When a source doc changes, agentpack knows. It compares recorded source hashes against the current files and marks affected skills as **stale**. Downstream skills that depend on a stale skill are marked **affected**. You always know what needs attention.

```
knowledge/selling-points.md  (changed)
        |
        v
@acme/value-copywriting      (stale — source changed)
        |
        v
@acme/landing-page-writer    (affected — depends on stale skill)
```

<Tip>
  Commit `.agentpack/compiled.json` to your repo so staleness detection works across CI, teammates, and fresh clones.
</Tip>
