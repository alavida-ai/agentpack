## Purpose

Document the checked-in fixture repos used by the replacement skills lifecycle tests.

## Fixture Repos

### `test/fixtures/monorepo/`

Represents the authoring side.

Contains:

- `.agentpack/build-state.json`
- `.agentpack/catalog.json`
- governed knowledge files
- multiple packaged domain skills
- one packaged methodology dependency
- multiple workbench-local skills that point at packaged skills via `requires`

Primary purpose:

- prove authoring-side inspection
- prove stale detection
- prove deterministic authoring metadata generation
- provide local package sources for install tests
- prove workbench-implied dependency roots
- prove workbench root union and deduplication from multiple local skills
- prove multi-root install and shared-dependency uninstall behavior

### `test/fixtures/consumer/`

Represents the consumer side.

Contains:

- `package.json`
- empty `.agentpack/install.json`

Primary purpose:

- prove npm-backed install into a consumer repo
- prove materialization into `.claude/skills/` and `.agents/skills/`
- prove runtime state reporting
- prove uninstall and reconciliation

## Why Checked-In Fixtures

The replacement CLI is mostly about filesystem state and package-manager behavior.

Checked-in fixtures make it easier to:

- inspect the intended architecture
- review test changes in diffs
- debug regressions without reconstructing repo shape in code

## Demo Sandbox

`sandbox/acme-demo/` is not part of the primary automated fixture layer.

It is a git submodule pointing at a standalone demo repo used for:

- manual product demos
- local exploratory testing
- walkthroughs of `skills dev`, DAG visibility, and stale-state behavior

Fixtures remain the automated safety net. The Acme sandbox is the human-facing demo surface.

## Update Rules

When a command starts depending on new runtime or authoring state, update fixtures intentionally:

1. prefer modifying the smallest existing fixture that proves the behavior
2. create a new fixture repo only when the existing ones no longer model the scenario clearly
3. keep fixtures curated, not exhaustive
