Current Architecture Map

How agentpack is currently organized in code

Runtime Shape
CLI Entry

src/cli.js

Global flags, top-level error handling, command registration.

Command Surface

src/commands/skills.js

Large commander adapter for inspect, validate, dev, install, stale, env, dependencies, status, missing, uninstall.

Command Surface

src/commands/plugin.js

Commander adapter for inspect, validate, build, dev.

Core Domain Logic

src/lib/skills.js

Owns most of the system: parsing metadata, validation, dependency sync, install state, build-state, stale detection, env/status, linking.

Core Domain Logic

src/lib/plugins.js

Plugin-local bundle inspection, validation, build staging, vendoring, watch mode. Reuses skill parsing and dependency utilities.

Shared Infra

src/lib/context.js

Repo root and workbench discovery.

Shared Infra

src/utils/*

Error taxonomy and output formatting.

Observed Structural Pattern

What is good

  • Very small surface area: only a few source files.
  • Commands map cleanly to user workflows.
  • Integration tests appear to drive behavior.

Maintenance risk

  • src/lib/skills.js is the dominant module and mixes multiple concerns.
  • Command adapters contain a lot of formatting logic.
  • Context, filesystem, lifecycle, and graph logic are not strongly separated yet.

Likely Responsibility Clusters Inside skills.js

A

Authoring Lifecycle

inspect, validate, stale, build-state, catalog

B

Consumer Install Lifecycle

install, uninstall, env, status, outdated, missing

C

Runtime Materialization

linking, syncing, local discovery paths, install state reconciliation

D

Dependency Graph

requires parsing, reverse deps, affected/stale status propagation
