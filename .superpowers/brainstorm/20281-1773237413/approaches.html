Implementation Approaches

Three ways to build the redesigned skill graph dashboard. Click the one you prefer.

A

Incremental Enhancement (Recommended)

Evolve the existing React + D3 stack. Replace the hardcoded 3-column layout with a proper D3 tree layout engine. Add routing with simple hash-based URLs. Restyle everything in place.

Pros

  • No new dependencies beyond what's already installed
  • Reuses existing server, watcher, and bundling infrastructure
  • D3's tree layout handles the DAG positioning natively
  • Smallest diff — less risk of breaking the working watcher/action flow

Cons

  • D3's imperative style means more manual DOM management
  • Hash routing is basic (no proper history API)
B

D3 Force-Directed + History API

Same React + D3 stack, but use a force-directed layout instead of a tree layout. Nodes auto-arrange based on their connections. Add proper History API routing for clean URLs.

Pros

  • Force layout handles complex graphs with cross-dependencies gracefully
  • Clean URL paths (/skill/name) instead of hash fragments
  • Looks more organic and dynamic

Cons

  • Force layouts don't enforce top-down directionality well — conflicts with the DAG requirement
  • Needs server-side route handling for history API
  • Physics simulation can feel jittery with frequent model updates
C

Canvas-Based with ELK Layout

Replace D3 SVG rendering with HTML Canvas for performance. Use ELK.js (Eclipse Layout Kernel) for proper layered DAG positioning. New dependency but purpose-built for directed graphs.

Pros

  • ELK produces the cleanest layered DAG layouts — exactly what we want
  • Canvas scales to hundreds of nodes without SVG DOM overhead
  • Professional-grade graph rendering

Cons

  • New dependency (elkjs ~2MB)
  • Canvas hit-testing for clicks/hover is more work than SVG
  • Bigger rewrite — replaces the entire rendering layer
  • Over-engineered for the likely graph sizes (tens of nodes, not thousands)
