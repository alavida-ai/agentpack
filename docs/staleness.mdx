A skill is only as good as the knowledge it was built from. If the source documents change but the skill is not re-authored, the agent works from outdated instructions. Staleness detection is how agentpack prevents that.

## The Core Idea

When you validate a skill, agentpack records a hash of every source file listed in `metadata.sources`. These hashes are stored in `.agentpack/build-state.json` alongside the skill's package name and the timestamp of the validation.

Later, when you run `agentpack skills stale`, agentpack re-hashes those same source files and compares. If any hash differs from what was recorded, the skill is stale -- it was built from knowledge that has since changed.

```
At validation time:            Later:
  process-guide.md → abc123     process-guide.md → abc123  ✓ same
  best-practices.md → def456   best-practices.md → xyz789  ✗ changed!
                                                  → skill is STALE
```

This is the same principle behind any build system's incremental rebuild: if the inputs changed, the outputs may need to be regenerated.

## Three Levels of Staleness

Staleness is not binary. agentpack distinguishes three states that describe where the change is and how it affects the graph.

### Changed

A specific source file has been modified on disk since the last validation. This is the most granular signal -- it tells you exactly which input moved.

**Example**: You updated `messaging-guide.md` to reflect a new brand voice. The file's hash no longer matches what was recorded in build-state.json.

In the [dashboard](/dashboard), changed sources appear as **red diamonds with a pulse animation**.

### Stale

A skill has at least one changed source. The skill itself may still contain the old knowledge -- it has not been re-authored from the updated sources yet.

**Example**: The `value-copywriting` skill references `messaging-guide.md` as a source. That file changed, so `value-copywriting` is now stale.

In the dashboard, stale skills appear as **amber circles with a glowing pulse**.

### Affected

A skill's own sources have not changed, but one of its dependencies is stale or affected. The skill may be indirectly impacted by upstream changes.

**Example**: `website-copy` requires `value-copywriting`. When `value-copywriting` becomes stale, `website-copy` becomes affected -- its own sources are fine, but something it depends on has drifted.

In the dashboard, affected skills appear as **hollow circles with a purple ring**.

## Checking for Staleness

Run `agentpack skills stale` to see which skills have drifted:

```bash
# List all stale skills
agentpack skills stale

# Detailed view with per-source hash comparisons
agentpack skills stale --detail

# JSON output for CI pipelines
agentpack skills stale --json
```

The list mode shows each stale skill with the count of changed sources. The detail mode shows the specific files that changed, with both the recorded and current hashes.

## Fixing a Stale Skill

Staleness is a signal, not an automatic fix. agentpack tells you what changed but does not rewrite the skill for you. The workflow is:

<Steps>
  <Step title="Identify the drift">
    Run `agentpack skills stale --detail` to see exactly which sources changed for each stale skill.
  </Step>
  <Step title="Re-author the skill">
    Review the updated source documents. Then re-author the SKILL.md to reflect the current knowledge -- either manually or by asking an agent to rewrite it from the updated sources.
  </Step>
  <Step title="Validate">
    Run `agentpack skills validate path/to/skill`. This checks the updated skill's structure and records fresh source hashes in build-state.json. The skill returns to **current** status.
  </Step>
  <Step title="Commit the build state">
    Commit `.agentpack/build-state.json` so that staleness detection works consistently across machines, CI, and teammates.
  </Step>
</Steps>

## How It Appears in the Dashboard

The [dev dashboard](/dashboard) makes staleness visible in real time. While the dashboard is running:

1. All nodes start green if everything is current
2. Edit a source file in your editor
3. The source node turns red (changed) immediately
4. The skill that references it turns amber (stale)
5. Any skills that depend on the stale skill show a purple ring (affected)
6. Click any node to open the inspector and see the details
7. Click **Validate** in the control strip to re-validate and clear the staleness

This live feedback loop means you can see staleness as it happens, not after the fact.

## Build State Persistence

The file `.agentpack/build-state.json` is the record of what was true at the last validation. It contains:

- The package name and version of each validated skill
- The relative path and content hash of each source file
- The timestamp of the validation

<Warning>
If you do not commit `.agentpack/build-state.json`, staleness detection only works on the machine where validation last ran. For staleness to work in CI, across teammates' machines, or after a fresh clone, commit this file.
</Warning>

## Why This Matters

Agent skills encode organizational knowledge -- processes, methodologies, standards. That knowledge changes over time. A skill written three months ago from a process document that has been revised twice since then will give the agent instructions that no longer match reality.

Staleness detection closes this gap. It creates a verifiable link between the source knowledge and the compiled skill, and it tells you the moment that link breaks. You will never ship a skill built from outdated knowledge again.

<Tip>
Integrate `agentpack skills stale --json` into your CI pipeline to block merges when skills have drifted from their sources. This ensures that published skills always reflect current knowledge.
</Tip>
