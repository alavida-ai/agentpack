agentpack commands are designed for CI. Every command runs non-interactively, returns structured output, and uses deterministic exit codes. Wire them into your pipeline to catch problems before they ship.

## What to run in CI

| Check | Command | When to run |
|---|---|---|
| Skill structure is valid | `agentpack publish validate` | Every PR |
| No stale skills | `agentpack author stale` | Every PR |
| Publish on merge | `npm publish` | On merge to main |

## Validation in CI

Run `agentpack publish validate` on every pull request to catch broken skills early. It checks SKILL.md parsing, source file existence, dependency sync, and package metadata.

```yaml
- name: Validate skills
  run: agentpack publish validate
```

When run without a target, it validates every authored skill in the repository. Exit code `0` means all skills pass. Exit code `2` means at least one skill has validation issues.

## Stale detection in CI

Run `agentpack author stale` to catch cases where source knowledge docs changed but the skills built from them have not been updated.

```yaml
- name: Check for stale skills
  run: agentpack author stale
```

This is especially powerful on PRs that modify knowledge docs. If a PR updates `domains/brand/knowledge/tone-of-voice.md`, the stale check will flag every skill that lists it as a source.

<Note>
  Stale detection requires `.agentpack/compiled.json` to be committed to the repo. Without it, there are no recorded hashes to compare against.
</Note>

## Complete GitHub Actions workflow

This workflow validates skills on every PR and publishes on merge to main.

```yaml .github/workflows/skills.yml
name: Skills CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          registry-url: https://npm.pkg.github.com
          scope: '@acme'

      - name: Install dependencies
        run: npm ci
        env:
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_PACKAGES_TOKEN }}

      - name: Validate all skills
        run: agentpack publish validate

      - name: Check for stale skills
        run: agentpack author stale

  publish:
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    needs: validate
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          registry-url: https://npm.pkg.github.com
          scope: '@acme'

      - name: Install dependencies
        run: npm ci
        env:
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_PACKAGES_TOKEN }}

      - name: Publish changed skills
        run: npm publish -w domains/brand/skills/copywriting
        env:
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_PACKAGES_TOKEN }}
```

## JSON output for machine parsing

Every agentpack command supports `--json` for structured output. Use this when you need to parse results programmatically in CI.

<Tip>
  Use `--json` output to build custom CI logic. For example, parse the stale check output to post a PR comment listing affected skills and their changed sources.
</Tip>

```yaml
- name: Check stale skills (JSON)
  run: |
    STALE_OUTPUT=$(agentpack author stale --json)
    echo "$STALE_OUTPUT" | jq '.count'
```

## Exit codes

All agentpack commands use consistent exit codes:

| Code | Meaning |
|---|---|
| `0` | Success — all checks pass |
| `1` | General error |
| `2` | Usage or validation error |
| `3` | Network error |
| `4` | Not found |

Use exit codes directly in CI `if` conditions or shell scripts. Codes 3 and 4 are useful for distinguishing registry connectivity issues from missing packages.

<Note>
  `NODE_AUTH_TOKEN` is the environment variable that npm reads for registry authentication. In CI, set it to a GitHub personal access token (PAT) with `read:packages` and `write:packages` scopes — store this as a repository secret named `GITHUB_PACKAGES_TOKEN`. Do not use the built-in `GITHUB_TOKEN` for cross-repo package access, as it only has permission within the current repository.
</Note>

<Note>
  For private registries, make sure authentication is configured before running CI checks. See [Authentication](/authentication) for CI setup.
</Note>

## Recommended PR checks

<AccordionGroup>
  <Accordion title="Required: validate">
    Block merge if any skill fails structural validation. This catches missing fields, broken source references, and dependency drift.
  </Accordion>
  <Accordion title="Recommended: stale">
    Warn (or block) when a PR modifies source docs without updating the skills built from them. Prevents knowledge drift from shipping.
  </Accordion>
</AccordionGroup>

<Warning>
  The publish step should only run on merge to main, not on PRs. Use the `if` condition in the workflow to gate it.
</Warning>
